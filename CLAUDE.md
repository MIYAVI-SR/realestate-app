## Git 運用ルール

### 基本方針：コード変更のたびに GitHub へプッシュする

- **すべてのコード変更は即座にコミット＆プッシュ**する
- 「あとでまとめてプッシュ」は禁止。作業単位が小さくても毎回プッシュする
- ブランチは `main` を基本とし、大きな機能追加は `feature/xxx` ブランチで作業する

### コミット手順

```
git add <変更ファイル>
git commit -m "変更内容の要約"
git push origin <ブランチ名>
```

### コミットメッセージ規則

| プレフィックス | 用途 |
|----------------|------|
| `feat:` | 新機能追加 |
| `fix:` | バグ修正 |
| `refactor:` | リファクタリング |
| `style:` | UI/スタイル変更 |
| `docs:` | ドキュメント更新 |
| `chore:` | 設定・依存関係変更 |

例: `feat: 物件一覧ページを追加`

### ブランチ戦略

- `main` — 本番相当。常に動く状態を保つ
- `feature/xxx` — 機能開発ブランチ（完成後に main へ PR でマージ）
- 直接 `main` へのプッシュは小規模修正のみ許可

## 開発ルール

- コードを変更したら必ず動作確認してからプッシュする
- `console.log` や `print` のデバッグコードはコミット前に削除する
- シークレット情報（APIキー、パスワード等）は `.env` に格納し、`.gitignore` で除外する
- `.env` ファイルは **絶対に GitHub にプッシュしない**
